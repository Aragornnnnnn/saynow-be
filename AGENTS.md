# AGENTS.md

이 파일은 리포지토리 루트 기준 모든 하위 파일에 적용된다.

## 기본 원칙

- 사용자와의 커뮤니케이션, 작업 요약, PR 설명은 한국어로 작성한다.
- 작업 전 관련 파일을 먼저 읽고, 기존 구조와 패턴을 우선 따른다.
- 요청 범위를 벗어난 리팩터링, 포맷 변경, 의존성 추가는 하지 않는다.
- 사용자가 만든 변경사항을 되돌리지 않는다. 작업 중 예상치 못한 변경이 보이면 먼저 현재 상태를 확인하고 그 변경과 공존한다.
- 작업 완료 전에는 반드시 double check를 수행하고, 실제로 실행한 검증 명령과 결과를 보고한다.

## 프로젝트 개요

- Saynow 백엔드 서비스다.
- 기술 스택은 Java 21, Spring Boot 4.0.6, Gradle Kotlin DSL, Spring MVC, Spring Data JPA, PostgreSQL, Flyway다.
- 패키지 구조는 기능 기준으로 `scenario`, `practice`, `feedback`, `common` 영역을 유지한다.
- 운영 DB는 PostgreSQL 기준이다. 테스트에서는 H2 PostgreSQL 모드를 사용한다.

## 자주 쓰는 명령

- 전체 테스트: `./gradlew test`
- 애플리케이션 실행: `./gradlew bootRun`
- 로컬 프로필 실행이 필요하면 `SPRING_PROFILES_ACTIVE=local`을 사용한다.

## 코드 작성 규칙

- Java 코드는 현재 프로젝트의 Spring MVC, JPA, DTO, Service, Repository 패턴을 따른다.
- API JSON 필드는 camelCase를 사용하고, DB 컬럼은 snake_case를 사용한다.
- 엔티티에는 불필요한 setter를 추가하지 않는다. 상태 변경 메서드나 생성자를 통해 의도를 드러낸다.
- 예외는 `common.exception`의 기존 `ApiException`, `ErrorCode`, `GlobalExceptionHandler` 흐름을 우선 사용한다.
- 새 API를 추가할 때는 컨트롤러, DTO, 서비스, 테스트의 책임을 분리한다.

## 데이터베이스 규칙

- 스키마 변경은 JPA 자동 생성에 의존하지 말고 Flyway migration으로 관리한다.
- migration 파일은 `src/main/resources/db/migration`에 추가한다.
- PostgreSQL 타입을 기준으로 작성한다.
  - 자동 증가 PK: `BIGINT GENERATED BY DEFAULT AS IDENTITY`
  - 날짜/시간: `TIMESTAMP(6)`
  - 소수: `NUMERIC(precision, scale)`
- 기존 migration 파일은 이미 배포됐을 수 있으므로, 수정이 필요한 경우 새 migration 추가를 우선 검토한다.
- 운영 환경 변수, DB 비밀번호, Supabase 연결 정보 등 secret은 커밋하지 않는다.

## 테스트와 검증

- 구현 변경 후 기본 검증은 `./gradlew test`다.
- DB 스키마나 엔티티 변경 시 Flyway migration, JPA 매핑, 통합 테스트가 함께 맞는지 확인한다.
- API 동작을 바꾸면 관련 통합 테스트를 추가하거나 수정한다.
- 문서만 변경한 경우에도 코드 변경 여부와 관련 구현 영향 여부를 확인한다.

## 문서 작업

- 문서의 DB 기준은 PostgreSQL로 유지한다.
- API 명세와 구현이 충돌하면 충돌 지점을 명확히 보고하고, 사용자의 판단이 필요한 부분을 분리한다.
- 프로젝트 외부 문서는 사용자가 명시적으로 요청한 경우에만 수정한다.

## Git과 PR

- PR을 작성하기 전 프로젝트에 PR Template가 있는지 확인한다.
  - 확인 경로 예: `.github/pull_request_template.md`, `.github/PULL_REQUEST_TEMPLATE.md`, `.github/PULL_REQUEST_TEMPLATE/`
- PR Template가 있으면 해당 양식을 따른다.
- 브랜치명이 `{type}/{number}` 형태라면 커밋 메시지는 `{number} {commitType}: {한글 메시지}` 형식을 사용한다.
  - 예: `feat/10` 브랜치에서는 `10 feat: 게시글 작성 기능 개발`
- 커밋 전에는 `git status --short`로 변경 범위를 확인하고, 의도하지 않은 파일을 포함하지 않는다.
